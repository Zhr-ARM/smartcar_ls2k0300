# 十字状态机（文字叙述版，按当前代码）

## 1. 代码位置

状态机转移在 `project/code/driver/vision/vision_route_state_machine.cpp`，十字边界改写在 `project/code/driver/vision/vision_image_processor.cpp`，默认参数在 `project/code/driver/vision/vision_config.c`，网页调试字段在 `project/code/driver/vision/vision_transport.cpp`。

## 2. 状态定义与进入优先级

十字主状态是 `VISION_ROUTE_MAIN_CROSS`，子状态分为 `VISION_ROUTE_SUB_CROSS_1`、`VISION_ROUTE_SUB_CROSS_2`、`VISION_ROUTE_SUB_CROSS_3`。在 `NORMAL` 和 `STRAIGHT` 中，代码会优先检查十字入口条件，十字成立时先进入 `CROSS_1`。

## 3. 状态转移条件（变量级）

从 `NORMAL/STRAIGHT` 进入 `CROSS_1` 由 `cross_entry_ready(input)` 决定。要求 `route_cross_detection_enabled` 为真，左右下角点都找到、索引都有效，并且 `left_corner_post_frame_wall_rows` 与 `right_corner_post_frame_wall_rows` 都不小于 `route_cross_entry_corner_post_frame_wall_rows_min`（默认 5）。

从 `CROSS_1` 进入 `CROSS_2` 由 `cross_stage2_ready(input)` 决定。要求“任意一侧下角点已找到且 `corner_y >= 75`”。

从 `CROSS_2` 进入 `CROSS_3` 由 `cross_stage3_ready(input)` 决定。要求 `left_start_frame_wall_rows` 和 `right_start_frame_wall_rows` 都不小于 `route_cross_stage2_enter_start_frame_wall_rows_min`（默认 1）。

从 `CROSS_3` 退出回 `NORMAL` 由 `cross_exit_ready(input)` 决定。要求 `start_boundary_gap_x > 0` 且小于 `route_cross_exit_start_gap_x_max`（默认 155）。

## 4. 从原图边界到IPM输入的主流程

每帧先从 trace 结果提取 `left_regular_pts/right_regular_pts`，并复制到当前工作边界 `left_pts/right_pts`。随后检测左右下角点，若某侧下角点存在，则从该下角点向上构造辅助线（aux trace/regular），再对该侧边界做“下角点截断”，并缓存截断后的基底边界 `left_cross_base_pts/right_cross_base_pts`。这些字段会被写入 `route_input`，交给 `vision_route_state_machine_update` 得到本帧 `route_snapshot`。然后按子状态执行 `CROSS_1` / `CROSS_2` / `CROSS_3` 的边界改写。全部改写完成后，才把 `left_pts/right_pts` 复制并做 IPM 映射，所以十字补线和截断都在原图边界工作集里完成。

## 5. CROSS_1 当前处理逻辑

进入 `CROSS_1` 后，先执行一条硬约束：若某侧下角点没找到，则该侧边界直接置空（`left_num=0` 或 `right_num=0`），避免无角点情况下继续补线。

`CROSS_1` 内会持续维护“最后一次检测到的左右下角点坐标”。当任意一侧下角点 `y >= 75` 时，状态机会转入 `CROSS_2`，并冻结这组坐标供下一阶段使用。

若某侧 aux 已建立且 `*_cross_aux_regular_num > 0`，代码会在 aux trace 中寻找方向值为 5 的上角点。当前调用是 `find_nth_or_last_dir_point_on_trace(..., target_dir=5, nth_match=6, ...)`，语义是优先取第 6 个 `dir=5`；如果不足 6 个，就回退到最后一个 `dir=5`；如果一个都没有则失败。

上角点命中时，会以“下角点到上角点”生成桥接线（`build_line_points_between_with_y_step(..., y_step=2)`），再把 aux regular 在上角点之前截断，最后按“base + bridge + aux_tail”拼接并回写。这里的 base 优先用 `left_cross_base_pts/right_cross_base_pts`。

如果上角点仍然失败（即该侧没有任何 `dir=5` 命中），该侧会回退到对应 `cross_base`，不做桥接，避免错误补线进入后续 IPM。

## 6. CROSS_2 当前处理逻辑

`CROSS_2` 会把 `CROSS_1` 冻结下来的下角点当作起点，重新执行“aux -> 上角点搜索”。如果某侧找到上角点，则直接用该上角点的 `x` 向下竖直延伸到巡线起始行 `maze_start_row`，作为该侧边界；如果该侧没找到上角点，则该侧边界置空。

## 7. CROSS_3 当前处理逻辑

`CROSS_3` 使用原先的“固定锚点 + 跳变截断点”策略。左锚点是 `{15,100}`，右锚点是 `{145,100}`，并使用 `kCross2JumpXThresholdPx=15`、`kCross2CutForwardPoints=2`。代码先在 regular 边界找跳变截断点，再把锚点与截断点连线，最后拼接“bridge + tail”回写到 `left_pts/right_pts`。

## 8. 调试时建议重点关注字段

建议优先看 `route_main_state/route_sub_state`、`cross_lower_left/right_corner_*`、`cross_stage2_frozen_left/right_corner_*`、`cross_left/right_corner_post_frame_wall_rows`、`cross_start_boundary_gap_x`、`cross_left/right_aux_found`、`cross_left/right_aux_trace_dir`、`cross_left/right_upper_corner_*`。这些字段能快速区分是状态机判定问题，还是 `CROSS_1/CROSS_2/CROSS_3` 补线分支导致的问题。
