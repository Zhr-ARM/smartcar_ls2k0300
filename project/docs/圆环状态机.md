# 圆环状态机

## 1. 当前位置

当前圆环状态机是 `vision_route_state_machine.cpp` 中已实际生效的一部分，不是设计稿。

它的职责有两层：

1. 根据角点、直边、起始贴边连续行数判断当前是否处于左右圆环流程。
2. 把结果反馈给 `vision_image_processor`，决定何时触发圆环补线以及控制层该使用哪类参数包。

当前主状态为：

1. `NORMAL`
2. `STRAIGHT`
3. `CIRCLE_LEFT`
4. `CIRCLE_RIGHT`

其中圆环相关部分是：

1. 左圆环：`CIRCLE_LEFT_1 ~ CIRCLE_LEFT_6`
2. 右圆环：`CIRCLE_RIGHT_1 ~ CIRCLE_RIGHT_6`

## 2. 输入来源

圆环状态机的输入在 `vision_image_processor_process_step()` 中组装，主要来自：

1. 十字下角点检测结果
2. 原图边界直边检测结果
3. 左右起始贴边连续行数
4. 左右边界点数
5. 角点索引
6. 原始左右边界在角点附近的间距检查结果
7. 当前基础 `preferred_source`
8. 本帧编码器增量

要点是：

1. 入口角点当前复用了十字下角点检测结果 `g_cross_lower_*`。
2. 圆环入口除了“单侧角点 + 对侧直边”，还要求原始左右边界在角点附近留有足够横向间距。
3. `frame_encoder_delta` 当前只累计到 `encoder_since_state_enter`，没有参与圆环状态转移条件。

## 3. 入口条件

### 3.1 `NORMAL -> STRAIGHT`

满足以下条件进入直道态：

1. 左边界是直边
2. 右边界是直边
3. 左侧没有角点
4. 右侧没有角点

一旦不再满足上述条件，就退回 `NORMAL`。

### 3.2 `NORMAL -> CIRCLE_LEFT`

左圆环入口条件为：

1. 右边界检测为直边
2. 左侧检测到角点
3. 左角点 `y > 60`
4. 左角点索引有效
5. 右边界点数大于 `route_circle_entry_min_boundary_count`，默认 `20`
6. 左角点索引小于 `右边界点数 - route_circle_entry_corner_tail_margin`，默认尾部余量 `20`
7. `left_circle_entry_raw_gap_ok == true`

其中 `left_circle_entry_raw_gap_ok` 当前由 `circle_entry_raw_boundary_gap_ready(...)` 计算。

当前配置里这三个入口 gap 参数没有显式赋值，因此按 C 的零初始化生效为：

1. `route_circle_entry_corner_row_offset = 0`
2. `route_circle_entry_gap_check_rows = 0`，代码中会被 `max(..., 1)` 修正为检查 `1` 行
3. `route_circle_entry_min_raw_boundary_gap = 0`

因此当前实际效果更接近：

只检查角点所在那一行原始左右边界是否都存在，且右边界 `x` 大于左边界 `x`。

### 3.3 `NORMAL -> CIRCLE_RIGHT`

右圆环入口是左右镜像：

1. 左边界检测为直边
2. 右侧检测到角点
3. 右角点 `y > 60`
4. 右角点索引有效
5. 左边界点数大于 `20`
6. 右角点索引小于 `左边界点数 - 20`
7. `right_circle_entry_raw_gap_ok == true`

## 4. 子状态流转

### 4.1 左圆环

`CIRCLE_LEFT_1`

1. 进入左圆环后的初始状态。
2. 默认 `preferred_source = RIGHT`。
3. 当 `left_start_frame_wall_rows >= 30` 时进入 `CIRCLE_LEFT_2`。

`CIRCLE_LEFT_2`

1. 仍保持 `preferred_source = RIGHT`。
2. 当 `left_start_frame_wall_rows <= 0` 时进入 `CIRCLE_LEFT_3`。

`CIRCLE_LEFT_3`

1. 运行时把 `preferred_source` 改为 `LEFT`。
2. `vision_image_processor` 会生成一条从右侧固定起点连向左侧规则边界目标点的辅助线，并用它替换右边界。
3. 当 `right_start_frame_wall_rows > 70` 时进入 `CIRCLE_LEFT_4`。

`CIRCLE_LEFT_4`

1. 继续把 `preferred_source` 设为 `LEFT`。
2. 当右侧再次检测到角点时进入 `CIRCLE_LEFT_5`。

`CIRCLE_LEFT_5`

1. 继续把 `preferred_source` 设为 `LEFT`。
2. `vision_image_processor` 会再次生成右边界辅助线，这次起点优先取右侧角点，否则退化到右侧规则边界起点。
3. 当 `right_start_frame_wall_rows >= 30` 时进入 `CIRCLE_LEFT_6`。

`CIRCLE_LEFT_6`

1. 状态机本身不再改 `preferred_source`，因此回到该主状态默认值 `RIGHT`。
2. 迷宫法起始搜索行会被抬高到至少 `route_circle_stage6_maze_start_row`，默认 `70`。
3. 当左右两侧都重新检测到直边时，退出到 `NORMAL`。

### 4.2 右圆环

右圆环按左右镜像执行：

1. `CIRCLE_RIGHT_1/2` 默认 `preferred_source = LEFT`
2. `CIRCLE_RIGHT_3/4/5` 运行时切换为 `preferred_source = RIGHT`
3. `CIRCLE_RIGHT_6` 再回到该主状态默认值 `LEFT`

转移条件镜像如下：

1. `RIGHT_1 -> RIGHT_2`：`right_start_frame_wall_rows >= 30`
2. `RIGHT_2 -> RIGHT_3`：`right_start_frame_wall_rows <= 0`
3. `RIGHT_3 -> RIGHT_4`：`left_start_frame_wall_rows > 70`
4. `RIGHT_4 -> RIGHT_5`：检测到左角点
5. `RIGHT_5 -> RIGHT_6`：`left_start_frame_wall_rows >= 30`
6. `RIGHT_6 -> NORMAL`：左右两边都重新成为直边

## 5. 补线和 `preferred_source` 的关系

这里有一个很容易混淆的点：补线替换的是“边界”，`preferred_source` 影响的是“后续更偏向使用哪一侧平移中线”。

当前代码中的对应关系是：

1. 左圆环 `state3/state5`：替换右边界，但 `preferred_source` 设为 `LEFT`
2. 右圆环 `state3/state5`：替换左边界，但 `preferred_source` 设为 `RIGHT`

也就是说：

1. 被补的通常是圆环外侧那一边。
2. 被偏好的通常是圆环内侧那一边。

这和旧文档里“补哪边就偏好哪边”的表述并不一致，当前应以代码实现为准。

## 6. 对后级模块的影响

圆环状态机当前会直接影响三件事：

1. `vision_image_processor` 是否执行 `state3/state5` 补线。
2. IPM 阶段 `render_ipm_boundary_image_and_update_boundaries(...)` 选择哪一侧作为优先中线来源。
3. `line_follow_thread` 选择哪组控制参数。

控制层当前参数映射是：

1. `CIRCLE_*_1` -> `CIRCLE_ENTER`
2. `CIRCLE_*_2/3/4` -> `CIRCLE_INSIDE`
3. `CIRCLE_*_5/6` -> `CIRCLE_EXIT`

所以圆环状态机已经同时影响视觉补线和运动控制。

## 7. 当前限制

当前圆环状态机还有几个需要明确标注的现状：

1. 入口角点复用的是十字下角点检测结果，命名上还没完全和“圆环入口角点”解耦。
2. `left_has_frame_wall/right_has_frame_wall` 已作为输入传入，但当前圆环状态转移只真正使用了起始贴边连续行数，没有直接使用这两个布尔量。
3. `encoder_since_state_enter` 在网页端可见，但当前没参与圆环推进条件。
4. `CROSS` 枚举仍保留，但默认关闭，且当前没有实际转入十字态的主链逻辑。
