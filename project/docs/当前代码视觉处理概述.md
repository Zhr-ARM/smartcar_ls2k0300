# 当前代码视觉处理概述

```text
/dev/video0
----> vision_frame_capture.cpp
-------> 采集 UVC full BGR 最新帧
----------> vision_image_processor.cpp
-------------> full BGR -> 160x120 BGR / GRAY / RGB565
-------------> 全图 OTSU -> 160x120 binary_u8
-------------> 迷宫法或 8 邻域提取原图左右边界
-------------> 提取角点 / 直边 / 起始贴边连续行 / 圆环辅助线
-------------> 更新 vision_route_state_machine
-------------> 按状态机对子状态执行圆环补线
-------------> IPM(280x140) 边界 -> 平移中线 -> 曲率 / track point / line_error
----------> vision_infer_async.cpp
-------------> 异步红框检测
-------------> 生成 ROI 并回写 red_rect / roi
-------------> 在 proc BGR / GRAY 上叠加 ROI 框
----------> vision_transport.cpp
-------------> 逐飞助手发送 binary / gray / rgb565 + boundary
-------------> Web UDP 发送 gray / binary / rgb
-------------> Web TCP 发送性能 / 边界 / IPM / 状态机 / 控制量
----------> line_follow_thread.cpp
-------------> 读取 line_error / track point / route_state
-------------> 按 NORMAL / STRAIGHT / CIRCLE_* 切换控制参数
----------> screen_display_thread.cpp
-------------> 按 send_mode 显示 binary / gray / rgb565
-------------> 叠加左边界 / 均值中线 / 右边界和控制文本
```

## 1. 当前实际生效的视觉主链

当前主链是：

`main.cpp -> vision_thread -> vision_pipeline -> vision_image_processor -> vision_transport`

其中：

1. `vision_thread` 持续循环执行 `vision_pipeline_process_step()` 与 `vision_pipeline_send_step()`。
2. `vision_image_processor` 是当前巡线、状态机、IPM、`line_error` 的核心处理模块。
3. `vision_infer_async` 已接入主链，但属于异步辅助链，不参与主巡线闭环判定。
4. `screen_display_thread` 不参与算法计算，只读取缓存做板载显示。

## 2. 启动顺序与线程分工

`project/user/main.cpp` 中和视觉相关的实际启动顺序是：

1. 初始化逐飞助手 UDP：`vision_assistant_udp_init(...)`。
2. 初始化网页端 UDP/TCP 通道：`vision_transport_udp_init(...)`。
3. 初始化 ncnn 默认模型：`vision_infer_init_default_model(...)`。
4. 启动视觉线程：`vision_thread_init("/dev/video0", &ncnn, ncnn_ready)`。
5. 启动巡线控制线程：`line_follow_thread_init()`。
6. 若 `screen_display_enabled=true`，再启动 `screen_display_thread_init()`。
7. 最后由 `main.cpp` 下发 send mode、推理开关、ROI 抓图、去畸变、IPM、中线和 `line_error` 参数。

当前并发拆分如下：

1. `vision_frame_capture.cpp`：独立采图，缓存 full BGR 最新帧。
2. `vision_thread.cpp`：视觉主线程，负责每帧处理和发送。
3. `vision_infer_async.cpp`：异步推理线程，做红框检测和 ROI 生成。
4. `line_follow_thread.cpp`：读取视觉输出做并级控制。
5. `screen_display_thread.cpp`：20Hz 板载显示线程。

## 3. 输入分辨率与中间图像

当前代码中明确使用两套尺寸：

1. full 图：`UVC_WIDTH x UVC_HEIGHT` 的 BGR，相机原始帧，用于推理和 ROI 映射。
2. 处理图：`160 x 120`，用于巡线、状态机、屏显、网页图传基础图像。
3. IPM 图：`280 x 140`，用于逆透视边界、中线和曲率计算。

`vision_image_processor` 当前维护的关键缓存包括：

1. `bgr_full`
2. `bgr`
3. `gray`
4. `binary_u8`
5. `rgb565`
6. `ipm_bgr`
7. 原图左右边界、均值中线
8. IPM 左右边界、左右平移中线、选中中线曲率
9. 原图圆环补线辅助线
10. 红框与 ROI 状态

## 4. 每帧处理流程

### 4.1 预处理

`vision_pipeline_process_step()` 每帧固定先调用 `vision_image_processor_process_step()`，后者会做：

1. 等待新 full BGR 帧。
2. 可选去畸变。
3. 缩放/转换得到 `160x120` 的 `bgr`、`gray`、`rgb565`。
4. 全图 OTSU，得到 `binary_u8`。

当前会记录 `capture_wait_us`、`preprocess_us`、`otsu_us`、`maze_us`、`total_us` 等耗时。

### 4.2 原图边界提取

当前边界提取支持两种模式：

1. `VISION_TRACE_METHOD_MAZE`
2. `VISION_TRACE_METHOD_EIGHT_NEIGHBOR`

默认配置 `maze_trace_method = 1`，也就是当前默认走 8 邻域边界跟踪。

这一步输出：

1. 原图左边界
2. 原图右边界
3. 原图左右均值中线
4. 左右边界点数
5. 八邻域原始 trace 点列和 `dir`

### 4.3 当前已接入的视觉语义特征

边界提取之后，当前代码会继续派生这些状态量：

1. 十字下角点
2. 原图边界角点
3. 原图边界是否为直边
4. 起始贴边连续行数
5. trace 是否碰到 frame wall
6. 圆环补线辅助线
7. IPM 边界角点与直边状态

其中最重要的两个事实是：

1. 角点和直边检测现在是并行输出，不是二选一关系。
2. 这些状态量会直接送入 `vision_route_state_machine_update(...)`，影响圆环状态与 `preferred_source`。

### 4.4 当前状态机的真实地位

当前状态机已经是主链的一部分，不再是占位实现。

当前主状态枚举为：

1. `NORMAL`
2. `STRAIGHT`
3. `CIRCLE_LEFT`
4. `CIRCLE_RIGHT`
5. `CROSS`

但当前实际生效的是前四类：

1. `STRAIGHT` 有真实进入/退出逻辑。
2. 左右圆环各有 `1~6` 六个子状态。
3. `CROSS` 枚举和子状态枚举仍保留，但默认配置 `route_cross_detection_enabled = false`，且当前主链没有进入十字态的转移实现。

### 4.5 圆环补线与状态联动

状态机更新后，`vision_image_processor` 会根据状态对子状态执行补线：

1. 左圆环 `state3`：生成右边界辅助线并替换右边界。
2. 左圆环 `state5`：再次生成右边界辅助线并替换右边界。
3. 右圆环按左右镜像，替换左边界。

补线结果不是只用于显示，而是继续进入后续：

1. 原图边界数组输出
2. IPM 映射
3. 平移中线生成
4. `line_error` 计算

### 4.6 IPM、中线和 `line_error`

如果 `enable_inverse_perspective=true`，当前链路会继续做：

1. 原图边界映射到 `280x140` 的 IPM 平面。
2. 对 IPM 边界做三角滤波、等距采样、角点/直边分析。
3. 基于左右边界法向平移，分别生成“偏左中线”和“偏右中线”。
4. 根据状态机给出的 `preferred_source` 和边界可用性，选择实际用于控制的一侧中线。
5. 计算 `track point`、曲率、平均绝对偏差和最终 `line_error`。

当前控制真正使用的是“选中的 IPM 平移中线”，不是原图均值中线。

`line_error` 当前支持三种取点方式：

1. 固定索引
2. 加权索引
3. 随速度索引

默认配置使用加权索引。

## 5. 推理链当前状态

`vision_infer_async` 已接入主链，但当前默认配置是：

1. `infer_enabled = false`
2. `roi_capture_mode = false`

所以默认上电时，这条链路处于“代码接入但功能关闭”状态。

推理打开后，它的实际作用是：

1. 提交 `160x120` proc BGR 和 full BGR 到异步线程。
2. 检测红框。
3. 生成 ROI。
4. 将 `red_rect` 与 `roi` 回写到 `vision_image_processor`。
5. 在 proc `bgr` 和 `gray` 图上画出 ROI 矩形。

即使推理关闭或无结果，也不会阻塞主巡线链。

## 6. 显示与外发

### 6.1 板载屏显示

`screen_display_thread.cpp` 当前会按 `send_mode` 显示：

1. `binary`
2. `gray`
3. `rgb565`

然后叠加：

1. 左边界红线
2. 均值中线黄线
3. 右边界蓝线
4. 电压
5. `Err`
6. 左右目标轮速
7. 转向输出
8. 采样行信息

当前不显示：

1. IPM 图
2. ROI 框
3. 状态机主/子状态

### 6.2 逐飞助手与网页端

当前外发分两类：

1. 逐飞助手 UDP：发送 `binary / gray / rgb565` 以及 boundary 包。
2. 网页端：
   - UDP 发送 gray / binary / rgb 图像
   - TCP 发送状态 JSON

TCP 当前已经接入的关键调试字段包括：

1. `line_error`
2. OTSU 阈值
3. 原图左右边界
4. IPM 左右边界
5. 选中的 IPM / 原图平移中线
6. `ipm_track_point`
7. `route_main_state`
8. `route_sub_state`
9. `route_preferred_source`
10. 圆环相关配置项和起始贴边连续行数
11. 电机目标速度和基础速度

## 7. 当前状态小结

当前“实际生效”的视觉链路可以概括成一句话：

相机 full BGR 输入后，先在 `vision_image_processor` 内完成原图边界、角点/直边、圆环状态机、补线、IPM 和 `line_error`，再把结果同时供给控制线程、板载屏和网页端。

还需要特别注意这几个当前状态：

1. 默认 trace 方法是 8 邻域，不是迷宫法。
2. 默认启用圆环状态机，默认关闭十字状态机。
3. 推理链代码已接入，但默认关闭。
4. 板载屏是边界调试视图，不是完整视觉 debug 面板。
5. 控制层已经按状态机主/子状态切不同参数包，不只是单纯读取一个 `line_error`。
