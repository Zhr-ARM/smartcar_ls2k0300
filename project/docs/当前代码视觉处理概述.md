# 当前代码视觉处理概述（按当前实际代码）

这份文档只描述当前代码里**真实存在且已接入主流程**的视觉处理链，目标是帮助你判断：

1. 哪些流程是控制闭环真正依赖的，不能直接删。
2. 哪些流程只是为了十字/圆环等特殊元素识别。
3. 哪些流程只是显示、图传、调试、抓图，理论上可独立裁剪。
4. 哪些流程虽然代码存在，但默认关闭，删前应先确认以后是否还要用。

---

## 1. 视觉总链路

当前实际主链是：

```text
main.cpp
 -> vision_thread_init("/dev/video0")
 -> vision_pipeline_init()
    -> vision_image_processor_init()
    -> vision_infer_async_init()
    -> vision_transport_init()

vision_thread 循环
 -> vision_pipeline_process_step()
    -> vision_image_processor_process_step()
       -> 等待采图线程给出最新 full BGR
       -> 可选去畸变
       -> 生成 bgr / gray
       -> OTSU + binary_u8
       -> 起点搜索
       -> 边界追踪（当前默认八邻域）
       -> 原图/规则边界提取
       -> 十字下角点 / 贴边 / 直边等状态量提取
       -> route_state_machine_update()
       -> 按状态机结果做 CROSS / CIRCLE 边界修正
       -> IPM 边界、中线、track_point、line_error
    -> 可选 vision_infer_async
       -> 红框 / ROI / ncnn 分类
       -> 结果回写 image_processor 缓存
 -> vision_pipeline_send_step()
    -> vision_transport_send_step()

line_follow_thread 循环
 -> 读取 line_error / track_point / route_main_state / route_sub_state
 -> 生成左右轮目标
```

结论：

1. **真正的控制主链**是 `vision_image_processor -> line_follow_thread`。
2. `vision_infer_async` 不决定巡线是否继续，只是附加功能。
3. `vision_transport`、`screen_display_thread`、`assistant_udp` 都不在控制闭环里。

## 2. 模块分工与真实作用

### 2.1 `vision_frame_capture.cpp`

作用：

1. 独立采图线程。
2. 持续等待底层 `wait_image_refresh()`。
3. 把最新一帧拷贝到 full BGR 缓冲。
4. `vision_image_processor_process_step()` 每次只取“最新一帧”。

实际地位：

1. 这是视觉主链入口，不能删。
2. 当前 full 图分辨率就是 `160x120`，处理链没有再做降采样。

### 2.2 `vision_image_processor.cpp`

作用：

1. 真正的视觉核心。
2. 包含预处理、二值化、边界追踪、十字/圆环识别配套信息提取、状态机输入组装、IPM、中线、`line_error`。
3. 保存大量调试缓存，供网页/屏幕读取。

实际地位：

1. 控制闭环最核心模块。
2. 如果要删流程，优先在这里区分“控制必需”和“调试缓存”。

### 2.3 `vision_route_state_machine.cpp`

作用：

1. 根据当前帧提取到的视觉特征，在 `NORMAL / STRAIGHT / CROSS / CIRCLE_LEFT / CIRCLE_RIGHT` 间切换。
2. 状态机会反过来影响 `vision_image_processor` 后续对边界的修正方式。
3. `line_follow_thread` 也会根据状态切换 PID 参数包。

实际地位：

1. 它不是单纯“上报状态”，而是同时影响视觉和控制。
2. 如果删掉十字/圆环/直道状态，需要同时改视觉修边逻辑和控制参数选择逻辑。

### 2.4 `vision_infer_async.cpp`

作用：

1. 异步红框检测。
2. 可选 ncnn 分类。
3. 可选 ROI 抓图。
4. ncnn 输入走 `BGR -> RGB` 转换后送入 `ncnn::Mat`，不使用 RGB565。
5. 可按分类结果动态修改 `ipm_center_target_offset_from_left_px`。

实际地位：

1. 默认关闭：`infer_enabled = false`、`ncnn_enabled = false`。
2. 关闭时不影响主巡线。
3. 属于明显可独立裁剪的分支。

### 2.5 `vision_transport.cpp` / `screen_display_thread.cpp`

作用：

1. 图像发送。
2. TCP 状态打包。
3. 调试图绘制。
4. 车载屏叠加显示。

实际地位：

1. 都不参与控制决策。
2. 删除风险主要在“失去调试观测能力”，而不是“车不走”。

## 3. 当前每帧真实处理顺序

下面是 `vision_image_processor_process_step()` 的实际顺序。

### 3.1 等待新帧

1. 调用 `vision_frame_capture_wait_next_bgr(...)`。
2. 超时则本帧失败，视觉线程本轮不继续。

### 3.2 预处理

1. full BGR 读入 `g_image_bgr_full`。
2. 若 `undistort_enabled=true`，执行 `cv::remap` 去畸变。
3. 否则直接拷贝到处理图 `g_image_bgr`。
4. BGR 转灰度 `g_image_gray`。
5. 保留处理分辨率 `g_image_bgr` 和 `g_image_gray`。

注意：

1. 当前已不再生成 `rgb565`。
2. ncnn 推理直接使用 BGR 图，不依赖车载显示格式。

### 3.3 斑马线检测（默认关闭）

条件：

1. `zebra_cross_detection_enabled = false` 时整个分支不跑。

打开后行为：

1. 在灰度图固定行采样。
2. 统计灰度跳变次数。
3. 命中则 `g_zebra_cross_count++`。
4. `main.cpp` 主循环里若计数达到阈值会直接停车。

结论：

1. 这不是巡线主链的一部分。
2. 默认关闭，明显可独立删除。

### 3.4 OTSU 二值化

当前行为：

1. 计算全局 OTSU 阈值。
2. 灰度图转二值图 `g_image_binary_u8`。
3. 做二值图滤波。
4. 绘制黑边框。

结论：

1. 这是后续边界追踪的直接输入，不能删。

### 3.5 起点搜索

行为：

1. 根据 `maze_start_row` 决定搜索起始行。
2. 结合上一帧左右起点位置做“带历史”的起点搜索。
3. 左右单边存在时会做边界身份纠正。
4. 若状态机处于圆环 `stage6`，会抬高搜索起始行。

结论：

1. 这是当前边界追踪稳定性的关键一环。
2. 不能直接删。

### 3.6 边界追踪

当前默认配置：

1. `maze_trace_method = 1`
2. 实际代表：**八邻域法**

实际做法：

1. 从左右起点分别追踪边界。
2. 保存原始 trace 点列和 dir。
3. 做一行一点提取，得到规则边界。
4. 统计起始贴边行数。
5. 统计原图直边。
6. 提取十字下角点等状态量。

结论：

1. 文档里如果还把当前主链叫“迷宫法”为主，需要改正。
2. 当前主流程的原始边界追踪核心是**八邻域法**。

### 3.7 十字相关特征提取

当前真实存在的十字配套特征有：

1. 下角点检测。
2. 双下角点配对有效性判断。
3. 起始贴边行数统计。
4. 角点后贴边行数统计。
5. 辅助边界构造。
6. 上角点检测。
7. `CROSS_1 / CROSS_2` 状态下的边界拼接、截断、补线。

这些不是纯调试，因为它们会参与：

1. `route_state_machine_update()` 的十字入态/出态判断。
2. 十字状态中的边界修正。
3. 最终 `line_error` 的计算结果。

结论：

1. 十字整套逻辑当前是**实打实接入主链**的。
2. 想删十字，不是删几个 debug 字段就行，而是要整块下掉：
   `十字特征提取 + 十字状态机条件 + 十字边界修正 + line_follow 对 CROSS 参数包支持`

### 3.8 圆环相关特征提取与补线

当前真实存在的圆环链包括：

1. 入环条件判断。
2. `CIRCLE_LEFT_1~6 / CIRCLE_RIGHT_1~6` 子状态推进。
3. `stage3 / stage5` 的单侧引导线构造。
4. `stage6` 抬高搜线起始行。
5. 圆环状态下动态改目标偏移中线位置。

这些同样不是纯调试，因为它们会直接改：

1. 当前使用哪条边界。
2. 边界如何补线。
3. IPM 中线偏移目标。
4. 控制线程选用的状态参数包。

结论：

1. 圆环逻辑当前也是真正接入控制闭环的。

### 3.9 状态机更新

`vision_route_state_machine_update()` 输入来自本帧视觉提取结果，核心判断包括：

1. `CROSS` 入态条件。
2. `CROSS_1 -> CROSS_2`。
3. `CROSS_2 -> NORMAL`。
4. 左/右圆环入态。
5. 圆环 1~6 子状态推进。
6. 直道 `STRAIGHT` 入态/退出。

当前默认配置下：

1. `route_cross_detection_enabled = true`
2. `route_circle_detection_enabled = true`
3. `route_straight_enter_consecutive_frames = 2`

说明：

1. 当前并不是只有 NORMAL/CIRCLE/CROSS。
2. `STRAIGHT` 也在真实生效，并且会影响控制参数包。

### 3.10 IPM、中线、track_point、line_error

最后阶段：

1. 将修正后的左右边界送入 IPM。
2. 生成左右平移中线。
3. 选择当前用于控制的中线来源。
4. 从所选中线计算 `track_point`。
5. 由 `vision_line_error_layer_compute_from_ipm_shifted_centerline(...)` 计算 `line_error`。

当前默认配置：

1. `enable_inverse_perspective = true`
2. `ipm_centerline_postprocess_enabled = true`
3. `ipm_centerline_triangle_filter_enabled = true`
4. `ipm_centerline_resample_enabled = true`
5. `ipm_centerline_curvature_enabled = true`
6. `ipm_line_error_method = 1`，即**加权索引**
7. 加权点默认 `{8, 12, 16}`，权重 `{0.5, 0.3, 0.2}`

结论：

1. 当前控制用的不是简单“某一行中点误差”。
2. 当前控制依赖的是 `IPM + 平移中线 + line_error_layer` 这一整层。

## 4. 控制闭环到底读了哪些视觉输出

`line_follow_thread.cpp` 当前真实读取的视觉量有：

1. `line_error`
2. `vision_image_processor_processed_frame_seq()`
3. `vision_image_processor_get_ipm_line_error_track_point(...)`
4. `vision_image_processor_route_main_state()`
5. `vision_image_processor_route_sub_state()`
6. `vision_image_processor_ipm_mean_abs_offset_error()`

因此，和控制直接相关的视觉输出至少包括：

1. `line_error`
2. `track_point`
3. `route_main_state`
4. `route_sub_state`
5. `mean_abs_path_error`

重要结论：

1. 当前车并不是只靠 `line_error` 在跑。
2. 如果删掉状态机，只保留 `line_error`，控制参数切换逻辑也得同步改。
3. 如果删掉 `track_point`，角速度环目标就没了，也得重写控制逻辑。

## 5. 当前流程按“控制必需 / 只剩调试 / 已弃用”重新分类

这一节是给“降 CPU 压力”直接用的。

### 5.1 控制闭环真正依赖的视觉结果

这些量当前仍然真正参与控制或状态切换，不能直接删：

1. 原图 `bgr/gray/binary_u8`。
2. 八邻域边界追踪与起点搜索。
3. 十字下角点、起始贴边、原图直边等**原图特征**。
4. 圆环原图入口判定相关特征。
5. `route_main_state / route_sub_state`。
6. IPM 后的左右平移中线。
7. `line_error`。
8. `track_point`。
9. `mean_abs_path_error`。
10. `selected_centerline_count / straight_required_last_index / straight_abs_error_sum`。

其中第 10 项虽然 `line_follow_thread` 不直接读 getter，但它们会先进入 `route_state_machine_update()`，决定 `STRAIGHT` 是否入态，所以仍然是控制依赖项。

### 5.2 仍在计算，但当前只剩网页/调试用途

这类最值得优先压缩，因为它们大多发生在 `IPM` 后，而且现在不再参与元素识别主决策。

#### A. `vision_line_error_layer.cpp` 中已删除的纯调试量

以下内容已经删除：

1. `selected_centerline_curvature` 曲率数组。
2. `ipm_weighted_decision_point`。
3. `src_weighted_decision_point`。
4. `ipm_weighted_first_point_error`。

代码证据：

1. 删除前，这些量的 getter 只被 `vision_transport.cpp` 读取。
2. `line_follow_thread.cpp` 不使用曲率，也不使用加权决策点。
3. `pid_tuning.h` 里的 `yaw_rate_ref_from_curvature_gain_dps` 字段已经不再接入控制。

结论：

1. 这部分删除后，不再为调试输出额外计算曲率和加权决策点。
2. `ipm_centerline_curvature_enabled` / `ipm_centerline_curvature_step` 目前只剩历史遗留配置语义，不再产生曲率数组输出。

#### B. `vision_image_processor.cpp` 中只给 `vision_transport.cpp` 的导出缓存

只被 `vision_transport.cpp` 读取，不参与控制：

1. 十字辅助边界 trace/regular 缓存。
2. 十字上角点缓存。
3. 十字 route debug 状态缓存。
4. 圆环 guide line 缓存。
5. 原图边界角点数组与状态导出。
6. IPM 边界角点数组与状态导出。
7. 原图/IPM 直边导出状态。
8. 左右 IPM/原图平移中线整条点列导出。
9. IPM 左右边界角度余弦数组导出。

注意：

1. “导出缓存只给 transport 用”不等于“其上游计算必然可删”。
2. 例如 IPM 角点检测本身目前还参与“首个角点截断后边界”的处理，所以**角点导出数组**可以删，但**角点检测/截断过程**不能直接判死。

#### C. 已删除的 IPM 彩色调试图

以下内容已经删除：

1. `g_image_ipm_bgr` 缓存。
2. `vision_image_processor_ipm_bgr_downsampled_image()` 接口。

结论：

1. `IPM` 彩色图缓存目前属于**真正无人消费**的死调试产物。
2. 这是一个明确的可删点。

### 5.3 默认关闭但代码仍在的分支

这些默认不影响当前运行结果，但保留着代码体积和少量维护成本：

1. `infer_enabled = false` 对应整条推理链。
2. `ncnn_enabled = false` 对应 ncnn 分类。
3. `zebra_cross_detection_enabled = false` 对应斑马线停车。
4. `screen_display_enabled = false` 对应车载屏显示。
5. `client_sender_enabled = false` 对应逐飞客户端发送。
6. `assistant_udp_enabled = false` 对应独立助手 UDP。

### 5.4 已经明确弃用或实质失效的代码/参数

这类可以视为“死代码”或“历史遗留参数”。

#### A. 已弃用代码

1. `vision_image_processor.cpp` 中的 `apply_circle_guide_line_to_side(...)`：
   当前编译器已提示 `defined but not used`。
2. `vision_image_processor.cpp` 中的 `find_cross_upper_corner_from_trace_after_frame_prefix(...)`：
   当前编译器已提示 `defined but not used`。
3. `RGB565` 生成与发送链：
   已从运行路径、枚举兼容层和配置说明中清理。
4. `roi_capture_mode` 抓图模式相关流程、变量与接口：
   已从推理链、pipeline、vision_thread、main 配置下发中清理。

#### B. 已弃用参数

1. `vision_processor_config_t::demand_otsu_keep_full_binary_cache`
   只在配置定义里出现，代码中无读取点。
2. `vision_runtime_config_t::circle_guide_anchor_offset_stage5`

   当前仅在 `pid_tuning.h` 里赋值或注释说明，控制逻辑不读取。

## 6. 针对你当前 CPU 压力的关键判断

你当前的状态是：

1. 常态 CPU > 90%。
2. 开启红框检测 + ncnn 时 CPU 100%。
3. 内存只有约 18%。

这意味着问题本质不是“内存不够”，而是“每帧算了太多现在并不需要的东西”。

### 6.1 不能简单把 CPU 压力“转移到内存”

在这个项目里，所谓“用内存换 CPU”只有这几种有效形式：

1. 预计算。
2. 缓存上一次结果，减少重复算。
3. 保存中间结果，避免同一帧里多次从头推导。
4. 直接不算某些只给调试看的量。

不能做到的事情：

1. 单纯“多开一点内存”不会让 `IPM`、角点、曲率、边界后处理自动变快。

### 6.2 你当前最值得做的方向

结合代码现状，CPU 优化重点不该放在“再改内存分配方式”，而该放在：

1. 停掉已经不参与控制的 `IPM` 后调试计算。
2. 缩减 `vision_transport` 依赖的调试导出。
3. 在 `IPM` 阶段只保留“控制最小闭环”需要的处理。

## 7. 从 CPU 优化角度看，建议的裁剪优先级

下面按“对控制影响最小、对 CPU 回报最大”排序。

### 7.1 第一优先级：删掉纯调试计算和无人消费产物

建议先动这些：

1. 已删除 `selected_centerline_curvature` 计算与导出。
2. 已删除 `ipm_weighted_decision_point / src_weighted_decision_point / weighted_first_point_error` 计算与导出。
3. 已删除 `g_image_ipm_bgr` 及其生成。
4. 删 `vision_transport.cpp` 中对十字辅助线、上角点、圆环 guide、原图/IPM 角点、左右整条中线数组、角度余弦数组的大量打包。

理由：

1. 这些东西当前不影响车跑。
2. 它们集中在 `IPM` 后半段和上位机输出路径，正好符合你说的“以前在 IPM 识别元素，现在改到原图后留下不少白算流程”。

### 7.2 第二优先级：把 `IPM` 后流程收缩成“最小控制闭环”

若目标是显著降 CPU，建议把 `IPM` 后阶段拆成两层：

1. 最小控制层：
   只保留 `IPM 变换 -> 左右平移中线 -> line_error -> track_point -> mean_abs_path_error -> straight判定量`。
2. 调试增强层：
   曲率、决策点、整条中线导出、角点导出、角度余弦导出等全部可选关闭。

这一步适合做成编译期开关或运行期开关。

### 7.3 第三优先级：评估是否弱化 IPM 角点支路

当前 `IPM` 后还在做：

1. 边界去重。
2. 回跳毛刺抑制。
3. 等距采样。
4. 三角滤波。
5. 三点法角度余弦。
6. 角点检测 + NMS。
7. 首角点截断。

这里面要分开看：

1. 前 1~4 项更像稳定中线本体的预处理。
2. 后 5~7 项原本更偏“IPM 特征识别”，但当前仍影响边界截断与中线质量。

建议：

1. 不要直接一刀切删掉整段角点支路。
2. 更合理的做法是先做 A/B 测试：
   保留去重/毛刺抑制/重采样，临时关闭“角点检测+首角点截断”，观察 `line_error` 稳定性、直道/弯道表现、十字/环岛通过率。

这一步是中风险优化，不建议作为第一刀。

### 7.4 第四优先级：再考虑是否削弱 `STRAIGHT` 相关 IPM 统计

当前 `STRAIGHT` 入态依赖：

1. `selected_centerline_count`
2. `straight_required_last_index`
3. `straight_abs_error_sum`

而 `line_follow_thread` 又使用：

1. `mean_abs_path_error`

这说明 `IPM` 不只是算 `line_error`，还在做速度规划和直道识别支撑。

建议：

1. 如果你后续决定取消 `STRAIGHT` 档、让直道和普通巡线统一参数，那么这些量可以继续往下裁。
2. 在目前结构下，不建议先删。

## 9. 建议的实际优化顺序

如果目标是“先明显降 CPU，再逐步验证赛道能力”，推荐顺序如下：

1. 先裁所有 `vision_transport.cpp` 相关的重型调试导出。
2. 再删曲率、决策点、IPM 彩色图这三类纯调试计算。
3. 再把 `IPM` 后处理拆成“控制最小集”和“调试增强集”。
4. 最后才测试是否可以去掉 `IPM` 角点检测 + 首角点截断。

这样做的好处：

1. 前两步对赛道行为影响最小。
2. CPU 收益通常最确定。
3. 一旦效果不够，再继续动中风险项。

## 10. 当前代码里几个容易误判的点

### 10.1 “元素识别改到原图了”不等于 “IPM 已经不重要”

现在十字/圆环的大量判定确实已经前移到原图，但 `IPM` 仍然负责：

1. 中线生成。
2. `line_error`。
3. `track_point`。
4. 直道判定支撑量。
5. 速度规划支撑量。

所以不能把整个 `IPM` 当成弃用流程。

### 10.2 “IPM 后很多量只剩调试用途” 这个判断是成立的

当前最明显的就是：

1. 曲率。
2. 加权决策点。
3. 大量角点/辅助线/整条中线导出。
4. `IPM` 彩色调试图。

这些正是当前最适合下刀的部分。

### 10.3 当前 CPU 过高更像“算法结构冗余”，不是“内存结构问题”

因为：

1. 内存只占约 18%。
2. 说明主问题不是缓存不够。
3. 而是视觉线程每帧计算路径过长，尤其是 `IPM` 后还保留了不少历史调试和旧特征流程。

---

## 11. 一句话总结

当前视觉代码可以分成三层：

1. **控制必需层**：原图边界追踪、原图元素状态、状态机、IPM 中线、`line_error`、`track_point`、直道/速度规划支撑量。
2. **仍在白算的调试层**：曲率、决策点、角点导出、辅助线导出、整条中线导出、IPM 彩色图。
3. **已弃用层**：少数死函数、死参数、已经不再接入控制的旧字段。

如果你的目标是尽快降低主板 CPU 压力，优先削第 2 层，再清第 3 层，最后才评估第 1 层里能不能继续缩到“最小控制闭环”。
