# 当前代码视觉处理概述

```text
/dev/video0
----> vision_frame_capture.cpp
-------> 采集 full BGR 帧（UVC 原始分辨率，缓存最新一帧）
----------> vision_image_processor.cpp
-------------> 去畸变 / 缩放到 160x120 / 保存 BGR、GRAY、RGB565
-------------> 全图 OTSU 二值化，得到 binary
-------------> 迷宫法或 8 邻域找左右边界，生成原图边界与均值中线
-------------> 识别十字下角点 / 环岛入口特征 / 边界直线性等状态特征
-------------> 逆透视 IPM（280x140），生成左右边界、平移中线、曲率、track point
-------------> 计算 line_error，输出给 line_follow_thread
-------------> 写出调试缓存：gray / binary / rgb565 / ipm / boundary / roi / red_rect
----------> vision_infer_async.cpp
-------------> 异步检测红框
-------------> 基于红框生成 ncnn ROI
-------------> 把 ROI 矩形回写到 BGR/GRAY 图像上
----------> vision_transport.cpp
-------------> 逐飞助手发送：灰度/二值/RGB565 + 边界
-------------> UDP Web 图传：gray / binary / rgb JPEG(PNG/BMP)
-------------> TCP Web 状态：性能、边界、IPM、角点、ROI、控制量
----------> screen_display_thread.cpp
-------------> IPS 屏显示图像底图
-------------> 叠加左边界/中线/右边界
-------------> 显示电压、误差、目标轮速、转向输出
----------> line_follow_thread.cpp
-------------> 读取 line_error、曲率、track point，参与速度与转向控制
```

## 1. 总览

当前工程的视觉处理主链由 `main.cpp -> vision_thread -> vision_pipeline -> vision_image_processor / vision_infer_async / vision_transport` 组成。

视觉模块的职责不是单一“巡线识别”，而是同时输出四类结果：

1. 控制量：`line_error`、曲率、跟踪点、赛道状态，供 `line_follow_thread` 使用。
2. 板载显示：给 `screen_display_thread` 提供灰度/二值/RGB565 图像与边界点。
3. 上位机图传：通过逐飞助手 UDP、网页 UDP/TCP 发送图像和元数据。
4. 推理辅助信息：异步红框检测结果、ncnn ROI、ROI 抓图。

## 2. 启动与线程分工

### 2.1 主入口

`project/user/main.cpp` 中和视觉直接相关的初始化顺序是：

1. 初始化逐飞助手独立 UDP 通道 `vision_assistant_udp_init(...)`。
2. 初始化网页端 UDP/TCP 通道 `vision_transport_udp_init(...)`。
3. 初始化 ncnn 模型 `vision_infer_init_default_model(...)`。
4. 启动视觉线程 `vision_thread_init("/dev/video0", &ncnn, ncnn_ready)`。
5. 按配置下发发送模式、推理开关、ROI 抓图、去畸变、IPM、中线后处理、line_error 计算参数。
6. 如果启用屏显，则启动 `screen_display_thread_init()`。

### 2.2 实际线程拆分

当前与视觉直接相关的并发执行有三层：

1. `vision_frame_capture.cpp`：采图线程，只负责等待 UVC 新帧并缓存最新 full BGR 图。
2. `vision_thread.cpp`：主视觉线程，循环执行 `vision_pipeline_process_step()` 和 `vision_pipeline_send_step()`。
3. `vision_infer_async.cpp`：异步推理线程，独立做红框检测和 ROI 生成。

另外还有一个独立的 `screen_display_thread.cpp`，它不参与算法计算，只消费视觉缓存做板载显示。

## 3. 输入与中间图像

### 3.1 输入

`vision_frame_capture_wait_next_bgr()` 从 UVC 驱动拿 full BGR 帧。当前处理链固定会保留：

1. full BGR：供异步推理与 ROI 映射使用。
2. proc BGR：下采样后的 160x120 BGR。

### 3.2 在 `vision_image_processor` 中维护的主要图像

从头文件和实现看，当前会维护这些可复用图像缓存：

1. `gray`：160x120 灰度图。
2. `binary_u8`：160x120 二值图，供巡线和二值显示。
3. `bgr`：160x120 BGR 图。
4. `bgr_full`：full 分辨率 BGR 图。
5. `rgb565`：160x120 RGB565 图。
6. `ipm_bgr`：280x140 的逆透视调试图。

这和 `project/docs/视觉流程.md` 中“保存原图像为 bgr/灰度/rgb565”的设计一致，但当前代码已经扩展到了 full BGR、binary、IPM、ROI 等更多缓存。

## 4. 主视觉处理链

### 4.1 预处理

`vision_image_processor_process_step()` 的前半段主要做：

1. 等待下一帧 full BGR。
2. 依据配置决定是否做去畸变。
3. 缩放/转换得到 `bgr`、`gray`、`rgb565`。
4. 对灰度图做全图 OTSU 二值化，得到 `binary_u8`。

代码里明确记录了这些阶段的耗时：

1. `capture_wait_us`
2. `preprocess_us`
3. `otsu_us`
4. `maze_us`
5. `total_us`

### 4.2 边界提取

当前主巡线边界提取有两种模式：

1. `VISION_TRACE_METHOD_MAZE`：默认模式，迷宫法。
2. `VISION_TRACE_METHOD_EIGHT_NEIGHBOR`：8 邻域边界跟踪。

边界提取的核心输出是：

1. 左边界 `x1/y1`
2. 中线 `x2/y2`
3. 右边界 `x3/y3`
4. 左右边界点数
5. 八邻域跟踪点与方向 `dir`

当前屏显、逐飞助手边界包、TCP debug 元数据都在复用这些数组。

### 4.3 特征识别与赛道语义

当前代码在边界提取后，还会从边界中派生一些“视觉语义特征”：

1. 十字下角点 `cross_lower_corner`
2. 原图直边判断 `src_boundary_straight_detected`
3. 左右边界是否撞到画面框 `left_has_frame_wall` / `right_has_frame_wall`
4. 左右边界起始贴边连续行数
5. 圆环补线辅助线
6. 边界首角点、角点索引、边界直线性

这些特征已经接入 `vision_image_processor` 的缓存与 TCP debug 输出。

其中原图直边判断与角点判断现在是并行关系，不互相覆盖：

1. 角点判断仍然使用八邻域 `dir` 模板识别十字下角点，当前模板已改为“相邻平台跳变”：
   前平台允许 `4/5` 或 `5/6`，短过渡允许 `3/4`，后平台允许 `1/2` 或 `2/3`。
2. 直边判断也使用八邻域 `dir`，在同一层读取原图边界前 `N` 个 `dir`，统计其中 `dir=4/5` 的占比。
3. 当前默认配置是检查前 `60` 个点，若其中 `90%` 以上为 `4/5`，则判为直边。
4. 这两个阈值都已配置化：`src_boundary_straight_check_count`、`src_boundary_straight_dir45_ratio_min`。
5. 角点和直边都不是直接对“未经处理的整条原始 trace”计算，而是先经过边框边界裁剪策略：
6. 若 trace 起始就贴着人工边框，且这段边框前缀较短，会先删除这段起始边框前缀。
7. 删除前缀后，如果后续再次碰到人工边框，则角点/直边判断只使用到“首次碰边框”为止的那一段 `dir`。
8. 只有一种例外：若起始边框段很短且被判定为可保留的初始 frame wall，则允许保留整段 trace 继续参与该层判断。
9. 角点定位优先落在过渡区中的 `3/4`，若没有过渡点，再回退到后平台中的 `2/3/1`。
10. 因此“检测到角点”和“检测到直边”在实现上不再互斥，是否采用由后续状态机自己决定。

### 4.4 当前“路由状态机”状态

这里有一个必须说明的现实情况：

`vision_route_state_machine.cpp` 当前实现是简化占位版。`vision_route_state_machine_update()` 每次都会把主状态重置回：

1. `VISION_ROUTE_MAIN_NORMAL`
2. `VISION_ROUTE_SUB_NONE`
3. `preferred_source = base_preferred_source`

也就是说：

1. 十字、环岛相关特征已经算出来了。
2. 这些特征已经被打包进 `route_input`。
3. 但当前状态机并没有真正把它们演化成复杂状态转移逻辑。

所以当前“视觉语义检测”是有的，但“基于语义驱动的路线状态机”还没有真正落地。

### 4.5 逆透视 IPM 与中线生成

如果 `enable_inverse_perspective` 打开，当前代码会继续做：

1. 将边界映射到 `280x140` 的 IPM 平面。
2. 对 IPM 边界做可选三角滤波。
3. 对 IPM 边界做可选等距重采样。
4. 基于左右边界法向平移，生成左右两条候选中线。
5. 对中线做可选后处理、滤波、重采样、曲率计算。

这里是当前控制使用的核心视觉处理，不再只依赖原图中线。

### 4.6 `line_error` 计算

`line_error` 当前来自 `vision_line_error_layer.cpp`，输入是“最终选中的 IPM 平移中线”。支持三种取点策略：

1. 固定索引 `FIXED_INDEX`
2. 加权索引 `WEIGHTED_INDEX`
3. 随速度索引 `SPEED_INDEX`

同时支持三种来源策略：

1. 偏左中线
2. 偏右中线
3. 自动选边

除了最终的 `line_error`，当前还会额外保存：

1. `ipm_track_index`
2. `ipm_track_point`
3. `ipm_weighted_decision_point`
4. `src_weighted_decision_point`
5. 选中中线的曲率序列
6. 整条路径平均绝对偏差

这些数据不仅用于控制，也会通过 TCP 元数据发给网页端。

## 5. 异步推理与 ROI

### 5.1 当前推理链路的实际作用

`vision_infer_async.cpp` 目前的主作用不是直接输出分类结果，而是：

1. 在处理分辨率图像上检测红框。
2. 依据红框位置生成 full 图 ROI。
3. 再把 full ROI 映射回 proc 坐标。
4. 将 ROI 矩形回写到 `vision_image_processor` 缓存。
5. 在 `bgr` 与 `gray` 图上直接画出 ROI 框。

所以当前推理链在可视化上最明显的表现是：

1. `red_rect`
2. `ncnn_roi`
3. ROI 抓图目录 `vision_roi_captures/`

### 5.2 当前状态说明

这条链路已经完全接入 `vision_pipeline_process_step()`，但它是“异步辅助链”，不会阻塞主巡线流程：

1. 推理关闭时，只清空红框和 ROI。
2. 推理无结果时，也不影响巡线主链继续运行。
3. 只有拿到异步结果时，才把 ROI 回写到图像缓存里。

## 6. 显示与外发

### 6.1 板载 IPS 屏显示

`screen_display_thread.cpp` 每 50ms 刷新一次，显示内容分两部分：

1. 图像底图：取当前发送模式对应的 `binary / gray / rgb565`。
2. 叠加信息：
   - 左边界：红色
   - 均值中线：黄色
   - 右边界：蓝色
   - 文本：电压、`Err`、左右目标计数、转向输出、采样行

它当前不显示：

1. IPM 图
2. 红框 ROI
3. 十字/环岛状态

也就是说板载屏显主要服务于“巡线边界调试”，不是完整 debug 面板。

### 6.2 逐飞助手发送

`vision_transport.cpp` 中的 client sender 会根据模式发送：

1. 灰度图
2. 二值图
3. RGB565 图

当模式为灰度时，还会额外发送边界包；非灰度模式下边界包会关闭。

### 6.3 网页端 UDP 图传

网页图传链会按配置决定是否发送：

1. gray 图
2. binary 图
3. rgb 图

编码格式支持 JPEG / PNG / BMP。UDP 侧采用分片包发送。

### 6.4 网页端 TCP 元数据

TCP 元数据是当前最完整的可视化调试出口，覆盖：

1. 性能数据：采图、预处理、OTSU、maze、总耗时
2. 图像参数：分辨率、阈值
3. 控制数据：`line_error`、基准速度、目标计数、滤波计数、电机输出
4. 红框与 ROI：`red_found`、`red_rect`、`roi`
5. 边界与 IPM：原图边界、IPM 边界、角点、角度 cos、中线、track point
6. 语义状态：十字下角点、环岛特征、route 状态

如果 `udp_web_data_profile` 不是 FULL，则只会发送一个精简状态集。

## 7. 视觉与控制的接口

视觉模块当前直接影响控制的接口主要有：

1. 全局 `line_error`
2. `vision_image_processor_processed_frame_seq()`
3. `vision_image_processor_get_ipm_selected_centerline_curvature(...)`
4. `vision_image_processor_get_ipm_line_error_track_point(...)`
5. `vision_image_processor_ipm_mean_abs_offset_error()`
6. `vision_image_processor_route_*()`

`line_follow_thread.cpp` 里明确把视觉看成“新帧驱动输入源”，按新视觉帧更新误差、预瞄曲率和速度缩放。

## 8. 当前代码里真正生效的“视觉处理”结论

如果只看当前已经接入主链且实际生效的部分，可以概括成：

1. 相机采图已经线程化，主视觉线程消费最新帧。
2. 灰度、RGB565、二值图都会实时生成并缓存。
3. 当前巡线核心仍然是 OTSU + 迷宫法/8邻域边界提取。
4. IPM、中线平移、曲率、`line_error` 已经是控制主输入。
5. 十字/环岛相关特征已在算，但 route 状态机本身目前还是占位实现。
6. 红框检测与 ncnn ROI 已接入，但属于异步辅助可视化链，不是主控制闭环核心。
7. 当前最完整的视觉观测出口不是板载屏，而是网页端 TCP debug 元数据。
