# Local Compute Receiver

这个工作区是给“少传输、网页端本地复算”路线准备的。

## 目标

只从车端接收：

- 原始帧：`gray` / `rgb` / `binary`
- 原始状态：时间戳、编码器转速、目标转速、基础速度、OTSU 阈值等

然后在浏览器本地用与车端一致的算法逐帧复算：

- 边界
- 角点
- 辅助线
- 平移中线
- `line_error`
- 各种调试图层

## 可行性结论

可行，但要区分两个层级：

### 1. 方向上完全可行

这条路线长期比“车端把所有数组都传给网页”更合理：

- 链路更轻
- 页面更灵活
- 复盘更容易
- 可以离线跑同一套视觉链

### 2. 如果目标是“和车端完全一样”

建议不要手写一份近似 JS 版本，否则很容易漂移。

最稳路径：

1. 把现有 C++ 视觉算法抽成纯计算层
2. 去掉硬件采图/发送依赖
3. 编译成 WebAssembly
4. 在浏览器 worker 中直接调用

这样网页端和车端才能最大程度共享同一套实现。

## 当前已完成

- 新建了独立工作区，不影响现有 `tools/pc_receiver_js`
- UDP 接收 `binary/gray/rgb` 三种 JPEG 模式
- TCP 接收 JSON 状态
- 网页能显示原始接收帧
- 浏览器 worker 能拿到“原始图像 + 原始状态”
- 本地复算接口已经搭好，后续算法可逐步迁入

## 当前限制

`public/pipeline_worker.js` 现在只是骨架，还没有把车端视觉链真正搬进去。

也就是说：

- 现在这个工作区证明了“架构和数据流是通的”
- 但还没有达到“效果和现有网页一模一样”

## 推荐下一步

优先迁移顺序：

1. OTSU / 二值化
2. maze 左右边界追踪
3. IPM 点变换
4. 边界去重 / 去毛刺 / 重采样
5. 角点检测
6. 辅助线生成与拼接
7. 平移中线
8. `line_error`

## 启动

```bash
cd tools/pc_receiver_local_compute
npm run dev
```

默认端口：

- UDP: `11000`
- TCP: `11001`
- HTTP: `8180`

可用环境变量覆盖：

- `BIND_HOST`
- `UDP_PORT`
- `TCP_PORT`
- `HTTP_PORT`
