# 总目标

## 目标总述

构建一套“单通道、双页面、同源算法”的网页调试与本地计算体系：

1. 主板只维护一条传输通道。
2. 同一条通道同时服务两个网页：
   - 完整调试页
   - 本地计算页
3. 通过配置参数决定主板发送哪一档数据：
   - `FULL`：发送完整调试结果
   - `RAW_MINIMAL`：只发送原始图像和原始采集值
4. 网页端行为与档位联动：
   - `RAW_MINIMAL`：只显示本地计算页
   - `FULL`：两个网页都可以显示
5. 本地计算页若有 `RGB` 则显示彩图，没有则不显示；但本地计算固定使用灰度图。
6. 最关键目标：
   网页端本地计算不是“近似模拟”，而是尽可能与车端一模一样。
   也就是：
   - 车端与网页端吃同类型输入
   - 车端与网页端尽可能跑同一份 C++ 算法源码
   - 主板编译一次，Web 再编译一次为 WASM
   - 输出的边界、中线、曲率、角点、辅助线、`line_error` 等尽可能一致

## 工程约束

1. 尽量不修改 `project/` 中现有算法文件。
2. 优先在 `tools/` 下完成桥接层、兼容层、网页端和 WASM 集成。
3. 如果后续某一步必须动 `project/` 里的现有算法文件，需要先确认再改。

## 当前阶段性成果

1. 单通道双页面服务已接好。
2. 主板配置项已支持 `FULL / RAW_MINIMAL / gray / rgb`。
3. 车端传输逻辑已支持两档状态发送和 `rgb jpeg`。
4. 网页端行为已按档位联动。
5. 两个网页已经共用接收层、格式化层、曲线图层、点线绘制层、像素探针层。
6. 本地计算页的交互壳已经具备。
7. 路线已经明确切换到“同源 WASM 算法”，不再把简化版 JS 识别当正式目标。
8. `wasm_bridge` 输出结构已经扩展到接近真实算法结果形态。

## 还未完成的核心工作

1. 只读桥接 `project/` 中真实视觉算法到 `tools/.../wasm_bridge`。
2. 为真实算法依赖补齐兼容桩。
3. 编译出第一版可调用的 WASM 产物。
4. 在浏览器 worker 中真正调用同源算法模块。
5. 让本地计算页输出结果与车端尽可能一致。

## 最终判定标准

如果网页端本地计算能够在输入同一帧图像和同一组原始采集值的前提下，
通过同源 C++ 算法的 WebAssembly 产物，输出与车端尽可能一致的边界、中线、曲率、角点、辅助线和 `line_error`，
则认为该总目标完成。
